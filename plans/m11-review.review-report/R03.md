## Detailed Findings

### R03-001: No lock held in _after_training but coordinator serializes

#### Background
Review of exception-safety paths in runtime hooks.

#### Expected behavior / invariant
If `sync_base_weights_to_active` raises, the `_actor_train_allocated` flag should remain True so `shutdown_hard` can retry the release.

#### Finding
The code at `miles_pipeline.py:608-610` calls `ray.get(coordinator.sync_base_weights_to_active.remote(...))` WITHOUT wrapping in try-finally. If this raises, the release call at line 614 is never executed. However, the `_actor_train_allocated` flag (set at line 578) remains True, and `shutdown_hard` (lines 667-680) retries the release unconditionally for any cluster_id marked as allocated. This is by-design per the comment "R11-F1: only flip the ledger flag after a SUCCESSFUL release" (line 612).

#### Evidence
Code lines 608-614 with comment line 612.

#### Impact
Correct by design.

#### Recommended action
No fix required.
