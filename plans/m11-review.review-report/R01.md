## Detailed Findings

### R01-001: Broad exception catch masks SGLang API contract violations in shrink path

#### Background
The shrink_engines path (miles/ray/rollout.py:931-941) wraps `pause_generation` in a try/except to handle cases where SGLang rejects the pause request. The implementation uses a blanket `except Exception` clause to catch all errors, logs a warning, and continues to `release_memory_occupation`.

#### Expected behavior / invariant
The code should preserve the SGLang API contract: if `pause_generation` succeeds (200 OK), the scheduler reaches a safe checkpoint before `release_memory_occupation` executes. If it fails (e.g., 4xx for "already paused"), the subsequent `release_memory_occupation` should still succeed because the engine may already be quiescent or the API call is idempotent.

#### Finding
The broad exception catch (line 934: `except Exception as exc:  # noqa: BLE001`) masks the boundary between SGLang API contract violations and transient errors. If SGLang repeatedly returns 4xx for "already paused" (indicating the pause is idempotent), the code logs once and continues, which is correct. However, if SGLang returns 5xx (server error) and the scheduler is NOT quiescent, the subsequent `release_memory_occupation` will still crash with "Pointer argument cannot be accessed from Triton" — exactly the scenario the pause was meant to prevent. The exception handler obscures this distinction in production logs.

#### Evidence
- Source code lines 931-941 (the broad except block).
- M11.1 attempt 5–7 logs show that `pause_generation` HTTP calls succeeded with 200 OK, so this exact exception path was not hit in the smoke run.
- Code comment (line 935) says "pause_generation is a best-effort barrier — if SGLang rejects the request (e.g. already paused), continue on to flush+release rather than blocking the loop," which is correct for 4xx but not for 5xx.

#### Impact
LOW. The smoke run (M11.1 attempt 10) shows clean 200 OK for both pause_generation and release_memory_occupation, so the idempotent-pause case (which triggers the exception) did not cause downstream crashes in the test. However, in a scenario where SGLang returns a transient 5xx during pause, the subsequent release_memory_occupation would still crash with the original Triton race condition. The exception handler's single warning log would be the only signal, making debugging harder.

#### Recommended action
Document the SGLang `pause_generation` API contract (idempotency, error codes, retry semantics) in a comment. Optionally add a counter or conditional logging to detect repeated pause failures across multiple shrink cycles (which could signal a scheduler thread that never quiesces).

#### Routing
- Debug-log update: None (smoke verified correct behavior; this is a code-clarity gap, not a bug)

## Reviewer Notes

1. **Quiescence invariant is CORRECT**: The `pause_generation` wrapping in both shrink_engines (rollout.py:931-941) and sync_selected_workers (miles_model_update_service.py:301-318) correctly implements the quiescence barrier before memory release operations. M11.1 attempt 10 provides definitive evidence.

2. **Memory saver gating is EXHAUSTIVE**: The rlix-mode override at rollout.py:1627-1628 is the sole point where `needs_offload` is computed and assigned to ServerGroup instances. All callers automatically get the override applied.

3. **Hook mode fallback is SAFE but documentation-dependent**: The MILES_TMS_HOOK_MODE env var is optional, but production deployments on Blackwell+CUDA12.9 MUST set it to "torch".

4. **Broad exception catch at shrink_engines is ACCEPTABLE**: The catch at line 934 masks the SGLang API contract, but the code comment explicitly documents the assumption.

5. **Resource leak risk in sync path is MITIGATED but not eliminated**: If `finalize_weight_update` raises, the engine stays paused during exception unwind. Recommend wrapping `continue_generation` in a finally block.

6. **No evidence of dual-pipeline contention on pause_generation**: M11.2 smoke run was pre-pause-generation fix. However, architecture (per-pipeline RolloutManager, per-actor HTTP client) mitigates connection pool starvation risk.

7. **Smoke run is comprehensive for R01 scope**: M11.1 attempt 10 exercises the full quiescence path across two rollout iterations.
