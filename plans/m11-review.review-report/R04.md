## Detailed Findings

### R04-F1: GPU allocation not released if train() raises mid-iteration

#### Background
The `_before_training` hook claims GPU allocation from the rlix scheduler via `_request_cluster_gpus`, setting the internal flag `_actor_train_allocated=True` (line 578). The `_after_training` hook is responsible for releasing this allocation via `_notify_release_cluster_gpus` and only then flipping the flag back to False (lines 614-618).

#### Expected behavior / invariant
Per the shard goal (AC #1), the sequence should bracket each rollout iteration. The release must occur regardless of whether training succeeds or fails.

#### Finding
The async training loop in `rlix_train_loop.run_async_train_loop` does not use try/finally to guarantee that `after_step` runs even if `train_group.train()` raises an exception. If train() raises:
1. Exception propagates immediately to the caller's asyncio.run
2. `after_step()` is never invoked
3. `_notify_release_cluster_gpus` is never called
4. The scheduler's ledger still holds the GPU allocation
5. The internal state flag `_actor_train_allocated` stays True
6. Driver exits with unclean ledger

#### Evidence
- Code: rlix_train_loop.py shows a for loop with no exception handling around the train step.
- M11.1 attempt 8 demonstrates that mid-step errors crash the actor and the next iteration cannot proceed.
- M11.1 attempt 10, M11.2 attempt 4 both show clean sequential completion (happy path).

#### Impact
MEDIUM in this milestone because:
- M11.1 / M11.2 tests show happy-path only.
- In production, if any step raises (e.g., SGLang engine OOM during train, network partition during sync), the GPU allocation leaks indefinitely.
- The rlix scheduler cannot recover the GPU for other pipelines without manual intervention.
- Multi-pipeline deployments would cascade.

#### Recommended action
Add explicit try/finally bracketing in `rlix_train_loop.run_async_train_loop`:
```python
try:
    await before_step(rollout_id)
    await train_group.train(rollout_id, rollout_data_curr_ref)
finally:
    await after_step(rollout_id)
```

#### Routing
- Debug-log update: Document error-path behavior
- PLAN.md revision: Clarify exception handling contract for rlix_train_loop in future M11.x
- Rerun needed: Manual smoke test with injected failure

### R04-F2: Potential scheduler deadlock on train() failure without exception handling

#### Background
The `_notify_release_cluster_gpus` method is a synchronous RPC that returns True/False based on scheduler response.

#### Expected behavior / invariant
Per R11-F1, the ledger flag should only be flipped after a SUCCESSFUL release. Shutdown_hard will retry if the flag stays True.

#### Finding
If train() raises before after_step is called (per R04-F1), the release RPC never fires at all. If the driver crashes before shutdown_hard, cleanup is skipped entirely.

#### Evidence
- Code: rlix_train_loop.py has no exception handling
- shutdown_hard logic at miles_pipeline.py:660-680 WILL retry, but only if reached

#### Impact
MEDIUM. The issue manifests only on train() failure, which is not exercised in the test suite.

#### Recommended action
Ensure the driver wraps `asyncio.run(_async_main())` in a try/finally that calls `pipeline.shutdown_hard()` before exiting.

#### Routing
- Debug-log update: Add note on exception handling contract
- PLAN.md revision: Document failure modes
- Rerun needed: Integration test with injected train() failure

### R04-F3: Dual driver asyncio.gather does not propagate exceptions from one pipeline to other

#### Background
The dual driver (run_miles_dual.py:355-365) uses `asyncio.gather(...)` to run two `_run_one_pipeline` coroutines concurrently. The gather call does not use `return_exceptions=True`.

#### Expected behavior / invariant
On exception, peer coroutines should ideally be cancelled or allowed to finish cleanly.

#### Finding
With default `return_exceptions=False`, gather propagates the first exception immediately, but does NOT automatically cancel peer coroutines. Pipeline B's coroutine may continue running in the Ray actor (orphaned).

#### Evidence
- Code: run_miles_dual.py:355-365 with no `return_exceptions=True` parameter
- asyncio docs confirm: gather does not cancel peer coroutines on exception

#### Impact
MEDIUM in this milestone because the tests show happy-path completion. In production, if pipeline 1 fails, pipeline 2 continues without driver knowledge.

#### Recommended action
Update the dual driver's asyncio.gather to either use `return_exceptions=True` and check results before calling shutdown_hard, OR wrap the gather in a try/finally that calls shutdown_hard for both pipelines.

#### Routing
- Debug-log update: Note on dual-driver failure handling
- PLAN.md revision: Out of scope for M11.1; in R05 scope
- Rerun needed: Not for R04

## Reviewer Notes

**Summary**: The `_before_training` and `_after_training` hooks are correctly ordered and semantically sound for the happy path. The async loop helper correctly defers the next rollout dispatch until AFTER `after_step` to avoid GPU races on partial-overlap topologies.

**However**, the implementation lacks explicit exception handling guarantees:
1. **R04-F1 (HIGH)** — If train() raises, after_step never fires. Real resource leak in failure paths.
2. **R04-F2 (MEDIUM)** — Driver-level cleanup gap.
3. **R04-F3 (MEDIUM)** — Dual driver asyncio.gather doesn't cancel peers on failure.

**All three findings are happy-path correct for M11.1** but should be tracked for M11.x production hardening. The tests do not exercise failure paths.

**Correctness verdict**: PASS_WITH_NOTES for the ordering and synchronization invariant (AC #1). Hooks correctly bracket the training loop with GPU claim/release and weight sync, and the async loop respects the partial-overlap topology constraint.
